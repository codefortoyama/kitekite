# 富山きてきて空港（β版）

富山空港を応援する**非公式**の私設ファンサイトです。
ビルド不要の HTML / CSS / JavaScript のみで構成され、GitHub Pages で公開しています。

- **公開URL**: https://codefortoyama.github.io/kitekite/
- **リポジトリ**: https://github.com/codefortoyama/kitekite
- **運営**: Code for Toyama City（有志）

> ⚠ 本サイトは富山空港・航空会社・行政機関・交通事業者とは関係のない非公式サイトです。
> 掲載の所要時間・運賃はサンプル（目安）です。正確な情報は各事業者の公式サイトをご確認ください。

## 1. ファイル構成

```
kitekite/
├── index.html          # トップページ（SEO / OGP / 構造化データ含む）
├── privacy.html        # プライバシーポリシー
├── guidelines.html     # 投稿ガイドライン
├── links.html          # 富山空港リンク集
├── css/
│   └── style.css       # 全ページ共通のスタイル
├── js/
│   ├── data.js         # データ・設定（投稿 / 比較データ / API設定）
│   └── main.js         # UIロジック（ナビ / 描画 / 絞り込み / ライブマップ）
├── img/                # 写真（ヒーロー背景・機材写真）
├── gas/
│   └── helpful-counter.gs  # 「役に立った」集計用 Apps Script（§10）
├── .nojekyll           # GitHub Pages で Jekyll 処理を無効化
└── README.md           # このファイル
```

**データとUIの分離**: 表示内容・設定は `js/data.js` に集約されています。

| 定数 | 内容 |
|---|---|
| `POSTS` | 掲載する投稿（管理者が承認したものを追記） |
| `CATEGORIES` | 投稿カテゴリー（8種） |
| `ROUTE_TABS` | 移動時間比較（東京→富山 / 富山→東京、各10スポット） |
| `FARES` | 運賃比較（購入パターン別6行） |
| `AIRPORT` | 富山空港の座標（ライブマップの中心） |
| `ADSB_SOURCES` | 機体位置データのAPIソース |
| `ROUTE_API` | 便名→経路の照会API（TOY発着判定用） |
| `GOOGLE_FORM` | 旧・直接POST用設定（現在は無効。§9参照） |
| `HELPFUL_API` | 「役に立った」集計APIのURL（未設定ならローカル動作） |

**外部依存**: Leaflet 1.9.4（地図描画・CDN読込・SRI付き）と Google アナリティクス（GA4・測定ID `G-0SFK86LPGH`・全ページに設置）。Leafletが読み込めない環境では静的地図に自動フォールバックします。

## 2. ローカルでの起動方法

環境構築は不要です。

```sh
git clone https://github.com/codefortoyama/kitekite.git
cd kitekite
```

- **最も簡単**: `index.html` をブラウザでダブルクリックして開く（file:// でも全機能動作します）
- ローカルサーバーを使う場合（任意）: `python -m http.server 8000` → http://localhost:8000

## 3. 公開と更新の方法

すでに **GitHub Pages（mainブランチ / ルート）で公開済み**です。更新は push するだけ:

```sh
git add -A
git commit -m "変更内容"
git push
```

push後 1〜2分で https://codefortoyama.github.io/kitekite/ に反映されます。

補足:

- canonical / OGP / JSON-LD のURLは公開URLに設定済み
- OGP画像（`ogp.png`・1200×630px推奨）は未配置。SNSシェア時のカード画像を出したい場合に追加する
- リポジトリは codefortoyama Organization 管理。メンバーに編集権限を付与して複数人運用が可能

## 5. 投稿の運用フロー

1. 利用者がサイトの埋め込みGoogleフォームから投稿
2. 管理者がGoogleフォームの「**回答**」タブで内容を確認（[投稿ガイドライン](guidelines.html)基準）
3. 掲載OKのものだけ `js/data.js` の `POSTS` に追記して push（そのまま自動公開はされない設計）

1件の形式（`data.js` 内にテンプレートコメントあり）:

```js
{
  id: 1,                     // 重複しない番号
  title: "投稿タイトル",
  category: "report",        // flight / access / gourmet / souvenir / sightsee / event / tips / report
  date: "2026-07-18",        // 掲載日または情報確認日
  author: "ニックネーム",
  body: "本文",
  image: null,               // 画像URL（null なら絵文字プレースホルダー）
  imageEmoji: "🏔️",
  url: "",                   // 参考URL（空なら非表示）
  helpful: 0,                // 「役に立った」初期値
  sample: false              // true で「サンプル投稿」バッジ表示
}
```

カテゴリーの読み替え: フライト情報→`flight`／空港アクセス→`access`／グルメ→`gourmet`／お土産→`souvenir`／周辺観光→`sightsee`／イベント→`event`／便利情報→`tips`／利用レポート→`report`

## 6. 移動時間・運賃の比較データ

`js/data.js` の `ROUTE_TABS`（各方向10スポット）と `FARES`（6パターン）で管理しています。

- 「東京 → 富山」: 富山県内の観光地へ／「富山 → 東京」: 首都圏のイベント会場・観光スポットへ
- 表示時に **飛行機有利 → 互角 → 新幹線有利** の順へ自動ソート（`main.js` の `sortByAdvantage`）
- 運賃はスポット別ではなく**購入パターン別**（変動が大きいため）。駐車場代の行のみ公式情報ベース（2026年7月確認）、他はサンプル表記

**所要時間・金額はサンプル（概算）です。** 定期的に各交通機関の公式情報と照らして見直してください。

## 7. ライブフライトマップの仕組み

Flightradar24は技術的（X-Frame-Options / CSP）にも規約的にも埋め込めないため、
**ADS-Bオープンデータ + Leaflet による自前ライブマップ**を実装しています（すべてAPIキー不要・無料）。

- 地図描画: [Leaflet 1.9.4](https://leafletjs.com/)（unpkg CDN・SRI付き）＋ OpenStreetMapタイル（初期ズーム: 7）
- 機体データ: `ADSB_SOURCES` に定義。**airplanes.live**（CORS対応を実測確認済み）を優先し、失敗時は adsb.fi を試行。取得範囲は空港から半径150NM（約280km）
- 経路判定: **adsbdb.com**（`ROUTE_API`）で便名から出発地・目的地を照会し、**TOY / RJNT 発着とみられる機体を赤色表示**。ポップアップに「HND → TOY」形式で経路も表示
  - 照会は**富山空港に近い機体から優先**、1回の更新につき最大8便・400ms間隔（レート制限への配慮）。結果はセッション内キャッシュ
- ステータス行: 総機数と「うち富山空港発着（確認分）」を表示。0機のときは「現在、このマップで確認できる富山便はありません」（**断定しない表現**にしている——運航の合間か、受信網の穴か、経路DB未登録かを画面からは区別できないため）
- 更新間隔: 60秒（ページ非表示中はスキップ）
- フォールバック: Leaflet読込失敗→静的OSM iframe／データ取得失敗→地図は残してステータス表示のみ／経路照会失敗→通常色のまま表示

運用上の注意:

- データは有志の受信ネットワーク由来のため**全機が写るとは限りません**（サイト上にも明記済み）
- 富山便は1日数往復（羽田線）のため、**飛んでいない時間帯があるのは正常**です
- 各APIのレート制限（目安: 1リクエスト/秒）を尊重し、更新間隔を極端に短くしないこと

## 8. 投稿フォームの受け口

### 現在の方式: Googleフォームの公式埋め込み（iframe）

- 回答の確認: Googleフォームの「回答」タブ（スプレッドシート連携も可）
- 質問の変更: Googleフォーム側で編集すれば、サイトは自動的に追従
- 表示の高さ調整: `css/style.css` の `.gform-frame`（PC: 1450px / スマホ: 1600px）
- 画像添付は不可（Googleフォームのファイルアップロードはログイン必須のため）。参考URLでの共有を案内済み

**重要な調査結果（2026年7月実測）**:

- 新しいGoogleフォーム（2026年版UI）は、外部サイトからの `formResponse` への直接POSTを**一切受け付けません**（entryパラメータ方式・セッショントークン付き・GET方式すべてHTTP 400）。ネット上の「自作フォーム→Googleフォーム送信」系の解説は旧仕様です
- 「ファイルのアップロード」質問が1つでもあると**フォーム全体がログイン必須**になります（外部からの匿名回答が不可になる）

### 自作フォームに戻したい場合（Formspree等）

検証・確認画面・二重送信防止・ハニーポット付きの自作フォームが `index.html` に `hidden` で温存してあります。

1. `index.html` の `.gform-wrap` / `.gform-note` と `<form id="post-form">` の `hidden` を入れ替える
2. `js/main.js` の **`submitPost(payload)` の中身だけ**を接続先に書き換える

```js
// Formspree の例
function submitPost(payload) {
  if (payload.honeypot) return Promise.resolve({ ok: true });
  return fetch("https://formspree.io/f/【フォームID】", {
    method: "POST",
    headers: { "Content-Type": "application/json", "Accept": "application/json" },
    body: JSON.stringify(payload)
  }).then(function (res) {
    if (!res.ok) throw new Error("送信失敗");
    return { ok: true, demo: false };
  });
}
```

Supabase接続の場合は `status: "pending"` 列を持つテーブル＋承認済みのみSELECT可のRLSを推奨（承認フローを維持するため）。

## 9. 「役に立った」ボタンの集計

- **現在はローカル動作**: 押した状態は閲覧者のブラウザ（localStorage）に保存。全訪問者の合計は集計されません
- **集計を有効にするには**（Googleスプレッドシート + Apps Script・無料）:
  1. [sheets.new](https://sheets.new) でスプレッドシートを作成
  2. 「拡張機能 → Apps Script」に `gas/helpful-counter.gs` を貼り付けて保存
  3. 「デプロイ → 新しいデプロイ → ウェブアプリ」（実行: 自分／アクセス: **全員**）
  4. 発行された `…/exec` URLを `js/data.js` の `HELPFUL_API` に貼って push
- カウントはシートに `[投稿ID, カウント]` で保存（手動修正・リセット可）。二重カウントはlocalStorageで防止（再押下で-1）
- Apps Scriptのコード修正後は「デプロイを管理 → 新バージョン」で**再デプロイ**が必要
